# Technical Specification (TZ)
**Project:** Telegram marketplace bot for selling one-time accounts + private web admin panel  
**Version:** MVP v1.1  
**Status:** Updated after clarification  
**Purpose:** Developer-ready specification for implementation

## 1. Project Overview
Build a Telegram bot that acts as a lightweight online marketplace for selling **one-time accounts** for different **platforms/plans**. Users browse products in Telegram, see price and product information, choose a payment method, and receive credentials after successful payment handling.

The system must include:
- a **Telegram bot** for customers,
- a **private web admin panel** for one admin,
- a **shared backend + database**,
- a **referral system** with internal balance,
- **manual screenshot review** for card/hybrid payments,
- **automatic delivery** for fully balance-covered orders.

The UI should remain minimal:
- Telegram side: inline keyboards only
- Admin side: simple private website, management-focused

---

## 2. Main Business Goals
1. Automate customer communication in Telegram without AI.
2. Show products, prices, stock count, FAQs, and extra info dynamically from database.
3. Accept payment screenshots for manual approval.
4. Deliver one account per order after successful approval/payment.
5. Give the admin a simple panel for inventory, pricing, texts, FAQs, orders, and reports.
6. Support Uzbek and Russian.
7. Support referral rewards and internal balance usage.

---

## 3. Roles

### 3.1 Customer
A Telegram user who can:
- start the bot,
- choose language,
- browse platforms/plans,
- view price, stock, FAQ, and extra info,
- place an order,
- choose payment method,
- submit a payment screenshot when needed,
- receive credentials,
- view balance and referral link,
- view order history,
- contact support/admin.

### 3.2 Admin
A single private admin who can:
- log into the admin panel,
- manage catalog and texts,
- manage credentials inventory,
- review screenshot-based payments,
- approve or reject orders,
- view reports/statistics,
- manage system messages,
- receive alerts about important events.

## 4. Product Model

### 4.1 Selling Model
- The system sells **accounts**.
- Each account can be sold **only once**.
- One order delivers **exactly one account**.
- Products are shown as a **full list** to users.

### 4.2 Structure
- A **platform** can have multiple **plans**.
- Example:
  - Platform: `Netflix`
  - Plans: `1 month`, `3 months`, `Premium`, etc.

### 4.3 Product Information Shown to User
For each plan/product, the bot shows:
- platform name,
- plan name,
- price,
- available stock count,
- description/info text,
- FAQ for that plan,
- out-of-stock state.

### 4.4 Product Media
- Text-only is enough for MVP.
- No product images required.

## 6. Payment Methods

## 6.1 Supported Methods
The bot must support 3 payment methods:
1. `Full card`
2. `From balance`
3. `Hybrid`

## 6.2 Full Card
- User pays full order amount by card.
- Bot shows fixed payment instructions.
- User sends screenshot.
- Order goes to admin review.
- Admin approves/rejects in web panel.

## 6.3 From Balance
- If user balance is enough to fully cover the product price:
  - order is auto-accepted,
  - no application is sent to admin for approval,
  - account is auto-delivered,
  - action must be logged.
- If balance is not enough:
  - method is unavailable,
  - bot asks user to choose another payment method.

## 6.4 Hybrid
- User balance is used as discount.
- Remaining amount is paid by card.
- Example:
  - Product: `70,000 UZS`
  - Balance: `10,000 UZS`
  - Card payment due: `60,000 UZS`
- Bot shows fixed payment instructions.
- User sends screenshot.
- Order goes to admin review.
- On rejection, reserved balance must be restored.

## 6.5 Payment Instructions
- Payment instructions are **fixed for everyone**.
- They should be editable from admin panel.
- They should support manual editing of all user-facing text variants.

---

## 7. Referral System and Balance

## 7.1 Referral Link
Each user has a unique referral link:
- `https://t.me/<bot_username>?start=ref_<code>`

## 7.2 Reward Logic
When a **new user** opens the bot via referral link and presses `/start`:
- inviter gets `+3,000 UZS`,
- reward is granted once for that invited user only,
- invited user must be a new bot user.

## 7.3 Referral Balance
Each user has internal balance in UZS.
Balance can only be used to pay for products inside the bot.

### Important rule
- Balance **cannot be withdrawn**.
- Balance is **spend-only** inside the bot.

## 7.4 Referral Abuse Protection
The system must block obvious abuse:
- no self-referrals,
- no duplicate reward for same invited user,
- repeated `/start` must not reward again,
- existing users cannot trigger referral reward later.

The system should also support basic suspicious referral detection.

If suspicious referral behavior is detected:
- bot/admin system should inform admin,
- the user who created that referral link should also be informed.

**Needs clarification:** exact suspicious behavior rules for MVP.  
**Recommended MVP scope:** basic alerts for repeated suspicious patterns, while still enforcing hard uniqueness rules.

## 7.5 Balance Usage Rules
- Balance can cover full price or partial price.
- In hybrid mode, available balance is used up to order total.
- Balance-only payments are auto-delivered and logged.
- Hybrid balance must be reserved until admin decision.
- On hybrid rejection/cancel, reserved balance must be restored.

---

## 8. Warranty and Support

## 8.1 Warranty Rule
- Warranty period: **7 days** after delivery.
- If user contacts regarding an order after 7 days, bot sends refusal text.

## 8.2 Support Handling
- Within 7 days: bot tells user to contact admin.
- After 7 days: bot shows refusal text.
- General support/contact option should still remain available.

## 8.3 Editable Messages
All such messages must be editable in admin panel, including:
- payment instructions,
- rejection messages,
- warranty-expired message,
- support texts,
- referral messages,
- other system texts shown to users.

## 9. User Flows

## 9.1 First Start Flow
1. User opens bot.
2. Bot registers Telegram user by `telegram_id`.
3. Bot asks for language if not chosen.
4. If referral parameter exists:
   - validate referral,
   - prevent abuse/duplicates/self-referral,
   - create referral relation,
   - add `3,000 UZS` to inviter if valid.
5. Show main menu.

### Suggested Main Menu
- Products
- My Orders
- My Balance
- Referral Link
- FAQ
- Support
- Language

---

## 9.2 Browse Products Flow
1. User opens `Products`.
2. Bot shows full list of platforms/plans.
3. User selects a plan.
4. Bot shows:
   - platform,
   - plan,
   - price,
   - available quantity,
   - description/info,
   - FAQ,
   - buy button.
5. If stock is zero:
   - show `Out of stock`,
   - disable buying.

---

## 9.3 Purchase Flow
1. User selects product.
2. Bot shows summary.
3. User chooses payment method:
   - Full card
   - From balance
   - Hybrid
4. Bot calculates order amounts.
5. Bot continues by selected method.

---

## 9.4 Full Card Flow
1. Create unpaid order.
2. Show fixed payment instructions.
3. User sends screenshot.
4. Bot saves screenshot and marks order as submitted.
5. Admin gets notification.
6. Admin reviews in web panel.
7. Admin approves or rejects.
8. If approved:
   - system assigns one available account,
   - sends `login + password`,
   - marks order delivered,
   - logs action.
9. If rejected:
   - bot sends editable rejection message.

---

## 9.5 Balance-Only Flow
1. Validate sufficient balance.
2. Create order.
3. Deduct balance.
4. Auto-assign one available account.
5. Send `login + password`.
6. Mark order delivered.
7. Log the action.
8. Admin does not receive approval application.

---

## 9.6 Hybrid Flow
1. Validate user has positive balance.
2. Reserve available balance up to order total.
3. Calculate remaining amount due by card.
4. Create order with:
   - original amount,
   - reserved balance,
   - card due amount.
5. Show fixed payment instructions.
6. User sends screenshot.
7. Admin reviews in web panel.
8. On approval:
   - reserved balance becomes final spend,
   - account is assigned,
   - credentials are sent.
9. On rejection:
   - reserved balance is restored,
   - bot sends rejection message.

---

## 9.7 User Order History
User can open `My Orders` and receive **all orders in one message**.

Each order entry must include:
- order ID,
- platform,
- plan,
- price,
- date,
- delivered login,
- delivered password.

### Important clarification applied
- Notes should **not** appear in order history.

---

## 9.8 Support Flow
1. User opens `Support`.
2. Bot offers general contact/admin access.
3. If user seeks help for a purchased order:
   - if within 7 days: tell user to contact admin,
   - if after 7 days: show refusal text.
4. General contact remains available even after 7 days.

---

## 9.9 Order Cancellation Flow
Users must be able to cancel unpaid orders themselves.

### Rules
- Only unpaid/unapproved orders can be cancelled by user.
- If order used reserved hybrid balance, reserved balance must be restored.
- If screenshot already submitted and order is under review, cancellation behavior should be defined.

**Assumption for MVP:** user can cancel only before screenshot submission.  
**Needs clarification:** whether cancellation is allowed after screenshot submission but before admin decision.

---

## 10. Admin Flows

## 10.1 Login
1. Admin opens private web panel.
2. Logs in with username/password.
3. Secure session is created.

## 10.2 Review Screenshot-Based Order
1. Admin opens `Orders`.
2. Sees pending list.
3. Opens order details:
   - order ID,
   - user info,
   - platform/plan,
   - original amount,
   - balance used,
   - remaining card amount,
   - screenshot,
   - timestamps.
4. Admin clicks `Approve` or `Reject`.

### Approve
- validate stock,
- atomically assign one available account,
- update statuses,
- send credentials to user,
- write audit log.

### Reject
- mark rejected,
- send editable rejection text,
- restore hybrid reserved balance if needed,
- write audit log.

## 10.3 Catalog Management
Admin can:
- create/edit/delete platforms,
- create/edit/delete plans,
- set prices,
- set active/inactive,
- edit multilingual descriptions,
- edit multilingual FAQ per plan,
- edit visible texts.

## 10.4 Inventory Management
Admin can:
- add credentials manually,
- bulk import credentials,
- edit optional internal notes,
- view stock counts,
- filter by platform/plan/status.

## 10.5 Settings Management
Admin can edit:
- payment instructions,
- rejection message,
- warranty-expired message,
- support text,
- referral texts,
- other user-facing system texts,
- referral reward amount if desired.

## 10.6 Reports and Analytics
Admin sees:
- total orders,
- paid/delivered orders,
- rejected orders,
- revenue,
- low stock,
- referral metrics,
- main charts.

## 17. Business Rules
1. One order = one account.
2. One account can be sold only once.
3. Stock count shown to users = count of available accounts only.
4. Out-of-stock products remain visible.
5. Balance is spend-only inside the bot.
6. Referral reward = `3,000 UZS` per valid new invited user.
7. Reward is granted on first valid `/start`.
8. No multiple simultaneous unpaid orders per user.
9. Balance-only orders are auto-delivered and logged.
10. Hybrid uses available balance up to order total.
11. Hybrid rejected/cancelled orders restore reserved balance.
12. Notes may be delivered with credentials internally if desired, but must not appear in user order history.
13. All user-facing texts must be editable from admin panel.
14. Support after 7 days for that order returns refusal text, but general support remains available.

---

## 18. Edge Cases and Error Handling

## 18.1 Stock
- If user tries to buy out-of-stock plan: show out-of-stock message.
- If admin approves when no stock exists: block approval and show error.
- Account assignment must be atomic to prevent duplicate delivery.

## 18.2 Payments
- If user uploads invalid file instead of screenshot: reject with instruction.
- If user submits screenshot with no active order: show error.
- If user tries balance-only without enough balance: disallow method.
- If user tries hybrid with zero balance: either disallow hybrid or treat it as full card.

**Assumption:** hybrid should require positive balance; otherwise it is not shown.

## 18.3 Referrals
- Self-referral: ignored and optionally logged.
- Duplicate referral: ignored and optionally logged.
- Existing user started via referral later: no reward.
- Suspicious referral patterns: create alert, notify admin, notify inviter.

## 18.4 Order Cancellation
- User can cancel unpaid order.
- If reserved hybrid balance exists, restore it.
- Cancelled order cannot later be paid/approved.

## 18.5 Warranty
- If user asks about product issue within 7 days: contact admin.
- If after 7 days: show editable refusal message.

## 18.6 Order History
- Only the owner of the Telegram account can access their history.
- History contains credentials, so access control must be strict.
